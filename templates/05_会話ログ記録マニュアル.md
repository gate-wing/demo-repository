# 会話ログの記録と再利用マニュアル
**Claudeとのやりとりを残して、次回から活用しましょう**

---

## STEP 1：会話を記録する

Claudeとの作業が一区切りついたら、以下のように伝えてください。

### そのまま使える指示フレーズ

**全文をそのまま残したいとき：**
```
ここまでの会話内容を一字一句変えずに conversation-2026-04-02.md というファイル名で保存して
```

**要点だけまとめてほしいとき：**
```
今日のやりとりを以下の形式でまとめて、conversation-2026-04-02.md として保存して

- 日付
- 業務テーマ
- やりとりの要点（箇条書き）
- うまくいったプロンプト
- 気づき・学び
```

**うまくいったプロンプトだけ残したいとき：**
```
今日のやりとりの中で、うまくいったプロンプトだけ抜き出してファイルに保存して
```

> **ポイント：** 日付部分は、その日の日付に変えてください。ファイル名は日本語を避けて、英数字にしましょう。

---

## STEP 2：プルリクエストを出して保存を確定する

Claudeがファイルを保存すると、「プルリクエスト（PR）」が自動で作られることがあります。
プルリクエストとは、**「この変更を正式に反映していいですか？」という確認の仕組み**です。

### 2-1. プルリクエストを確認する

Claudeが保存後にURLを教えてくれます（例：`https://github.com/gate-wing/demo-repository/pull/23`）。
そのURLをブラウザで開いてください。

もしURLがわからなくなったら：
1. GitHub のリポジトリページを開く
2. 上部の **[Pull requests]** タブをクリック
3. 自分の名前やファイル名が含まれるものを探して開く

### 2-2. 内容を確認する

1. PRのページで **[Files changed]** タブをクリックすると、変更内容が見られます
2. 追加された部分は **緑色**、削除された部分は **赤色** で表示されます
3. 内容に問題がなければ、次のステップへ進みましょう

### 2-3. マージする（変更を確定する）

1. PRのページに戻り、下の方にある **緑色の [Merge pull request] ボタン** をクリック
<img width="937" height="809" alt="image" src="https://github.com/user-attachments/assets/8241b471-30c7-487c-a954-f0d1c403b2f4" />

3. 確認画面が出たら **[Confirm merge]** をクリック
<img width="896" height="443" alt="image" src="https://github.com/user-attachments/assets/cd9334f7-8886-4b69-95fc-13c28271b0f8" />

✅ これで変更がリポジトリに正式に反映されました！

> **補足：** マージした後、紫色で「Merged」と表示されればOKです。ブランチの削除を聞かれたら **[Delete branch]** を押して構いません（不要になった下書きスペースの片付けです）。
<img width="941" height="364" alt="image" src="https://github.com/user-attachments/assets/926a1c51-c973-409e-bc53-b97bcb360358" />

### 2-4. 保存されたことを確認する

1. リポジトリのトップページに戻る（上部の **[<> Code]** タブをクリック）
2. `participants` → 自分のフォルダ を開く
3. 新しいファイルが追加されていれば OK

※ 保存先やファイル名が違っていたら、Claudeに「participants/（自分のフォルダ）/ に保存して」と伝え直せば大丈夫です。

---

## STEP 3：次回、過去の記録を活用する

新しくClaudeを開くと、前回の会話内容は残っていません。
GitHubに保存した記録をClaudeに読み込ませることで、前回の続きから作業できます。

### なぜ新しい会話で始めた方がいいのか

前回の会話をそのまま続けることもできますが、**こまめに新しい会話に切り替える**ことをおすすめします。

- **会話が長くなるとClaudeの回答の質が落ちます。** やりとりが増えるほど、前の方の内容を忘れたり、的外れな回答が出やすくなります
- **新しい会話でやりとりの要点だけ読み込ませると、Claudeは最高の状態で作業を始められます**
- 必要な情報だけを渡すので、余計なやりとりに引っ張られず、的確な回答が返ってきます

> **たとえるなら：** 散らかったデスクで作業を続けるより、一度片づけてから必要な資料だけ並べ直した方が、仕事がはかどるのと同じです。

### 3-1. リポジトリをClaudeに教える

新しい会話を始めたら、最初にリポジトリのURLを伝えます。

```
このリポジトリで作業したい
https://github.com/gate-wing/demo-repository
```

> Claudeがリポジトリの内容を把握してくれます。初回はClaudeから名前の確認があるので、答えてください。

### 3-2. 前回の記録を読み込む

自分のフォルダにある過去の記録を読んでもらいます。

```
participants/（自分のフォルダ）/ にあるファイルを確認して、前回の内容を把握して
```

特定のファイルを指定してもOKです。
```
participants/（自分のフォルダ）/conversation-2026-04-02.md を読んで
```

### 3-3. 前回の続きから作業する

記録を読み込んだ後、やりたいことを伝えましょう。

**続きをやりたいとき：**
```
前回の内容を踏まえて、同じ業務の続きを進めたい
```

**前回のプロンプトを改善したいとき：**
```
前回うまくいったプロンプトをベースに、もっと良くできる点があれば改善案を出して
```

**他の人の記録を参考にしたいとき：**
```
participants/ の中で、他の人がどんなプロンプトを作っているか見せて
```

### まとめ：次回の会話の始め方

| 順番 | やること | 伝えるフレーズ |
|:---:|---------|---------------|
| 1 | リポジトリを指定する | `https://github.com/gate-wing/demo-repository で作業したい` |
| 2 | 前回の記録を読み込む | `participants/（自分のフォルダ）/ のファイルを確認して` |
| 3 | やりたいことを伝える | `前回の続きをやりたい` など |

---

## 記録ファイルの命名ルール

| 種類 | ファイル名の例 |
|------|---------------|
| 日付ごとの記録 | `conversation-2026-04-02.md` |
| テーマ別の記録 | `conversation-weekly-report.md` |
| プロンプト集 | `prompts-email-templates.md` |

## よくある質問

**Q. 毎回記録しないとダメ？**
→ 必須ではありません。「これは残しておきたい」と思ったときだけでOKです。

**Q. 失敗したやりとりも残す意味はある？**
→ あります。「この指示だとうまくいかなかった」という記録は、次回の改善に役立ちます。

**Q. 他の人の記録を見てもいい？**
→ はい。`participants` フォルダ内の他の方のファイルも自由に見られます。他の人のプロンプトを参考にするのはとても効果的です。
